# BUGS

- Review note for `1461124`: this change only expands the Playwright/manual QA acceptance coverage and does not introduce a new product bug fix entry.

- `/app` initially rendered a demo-focused notes shell instead of the real org-management shell. Fixed in `dc71c83`.
- The first DB-backed notes path still mixed real queries with demo-only helpers, which would have broken configured mode. Fixed in `b5a11c4`.
- File and AI API routes initially trusted an `x-user-id` header instead of the authenticated session. Fixed in `b5a11c4`.
- Fresh local DB commands could miss `.env.local`, which made `db:generate`, `db:push`, `db:studio`, and `db:seed` depend on the shell environment instead of the project default. Fixed in `92c142e`.
- The seed path imported helpers marked `server-only`, which is safe for Next.js request handling but unsafe for CLI execution and can break seeding before any app code runs. Fixed in `92c142e`.
- Seeding a fresh Supabase project could fail while uploading seed files because the storage bucket was assumed to exist already. Fixed in `92c142e`.
- The DB-backed notes list query ordered by a literal fallback rank when no search term was present, which Drizzle compiled into `order by 0 desc` and caused `/app/notes` to crash for normal browsing. Fixed in `13e4990`.
- The first cursor-based `Load more` implementation built second-page SQL predicates with JS `Date` objects inside the query template, which caused `/api/notes` cursor requests to fail even though the first page loaded correctly. Fixed in `990f16e`.
- The earlier docs automation strategy relied on a nested post-commit Codex run, which made it possible to push code without updated `NOTES.md`, `AI_USAGE.md`, `BUGS.md`, and `REVIEW.md` when the refresh step hung or was bypassed. Fixed in `990f16e`.
- The notes page still split search behavior between a server-rendered top-level form and a client-side paginated feed, which left the UX inconsistent once pagination moved into the feed. Fixed in `ca54d49`.
- The note surfaces still had a few polish regressions after the pagination/search refactor: missing back navigation on edit, weak input/body contrast, and button labels that could wrap awkwardly in tighter layouts. Fixed in `364bb41`.
- The sign-up flow dropped entered `displayName` and `email` values whenever validation or Supabase returned an error, which forced users to retype those fields on every failed submission. Fixed in `6224c3a`.
- The paginated DB search path had regressed from earlier substring-style matching to full-text matching only, which meant some note queries no longer returned expected results after pagination/search moved into the API path. Fixed in `6224c3a`.
- The debounced notes search flow could briefly show the correct client-fetched results and then overwrite them with stale server-fed `initialItems` after `router.replace`, because the feed treated the URL-driven re-render as authoritative even when the client already owned fresher results. Fixed in `62a0574`.
- Accepting an AI summary left the accepted draft in local component state until the refresh completed, which could briefly show stale draft content and stale selection state even though the acceptance had already been applied. Fixed in `e3850a8`.
- Note tags were not constrained tightly enough at the form boundary, so users could submit too many tags or oversized tag strings and only learn about the problem late, with weaker validation feedback than intended. Fixed in `e3850a8`.
- The app-level email confirmation flow depended on dedicated `/auth/callback` and `/auth/confirmed` routes plus `emailRedirectTo`, which was more fragile than the rest of the auth setup and mismatched the simpler “instant access” local/dev flow. That app-managed confirmation path was removed, and signup now explicitly assumes Supabase email confirmation is disabled when immediate access is required. Fixed in `70f120a`.
- The auth pages redirected based only on the raw Supabase session, which meant users could be bounced into `/app` even when the app-specific auth context could not be built yet, and Railway health checks still hit `/` instead of a dedicated health endpoint. The app now uses a complete auth-context check for sign-in/sign-up redirects, exposes `/api/health`, and renders a controlled workspace error state for non-redirect shell failures. Fixed in `a8f307d`.
- Note create/edit still relied on redirect-style server actions for validation failures, which meant users could lose context and not get field-level feedback when note validation rejected the payload. The note actions now return structured action state, preserve typed values, and surface field errors inline. Fixed in `7db3bc6`.
- The paginated notes feed could append duplicate cards when overlapping page payloads or repeated client fetches reintroduced note ids that were already present in local state. The client feed now deduplicates appended note ids defensively before extending the list. Fixed in `7db3bc6`.
- The Docker builder stage did not receive the required public Supabase env values, which made production builds depend on those variables already existing in the build environment instead of being passed through explicitly. The builder now accepts and exports `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` during `pnpm build`. Fixed in `4419bb1`.
- Railway was still configured to build with Nixpacks even after the repo’s deployment path had been hardened around the checked-in Dockerfile, which meant hosted builds could diverge from the reviewed local container setup. Railway now uses `builder: DOCKERFILE` with the repo Dockerfile explicitly. Fixed in `d1a71a4`.
- The browser Supabase client imported public env from the server-side Zod env wrapper, which prevented Next.js from inlining `NEXT_PUBLIC_*` values reliably into the client bundle. The public browser env now lives in a dedicated helper with direct `process.env.NEXT_PUBLIC_*` reads. Fixed in `88c7c4b`.
- The selective-sharing note form still required users to paste raw org member UUIDs into `sharedUserIds`, which made the feature effectively unusable in normal UI flows even though the backend support existed. Note create/edit now loads organization members and renders a checkbox picker for sharing instead of a UUID text field. Fixed in `00b2ff6`.
- Navigating to a deleted or inaccessible note URL rendered the generic Next.js 404 page instead of a branded experience with a route back to the notes list. The note detail route now ships a dedicated not-found page with a clear return path. Fixed in `84b56f2`.
- The Delete button on the note detail page gave no feedback while the server action was running, leaving it looking interactive and potentially triggering double-clicks. The delete action now uses a client wrapper with `useFormStatus` to show a disabled `Deleting…` state. Fixed in `84b56f2`.
- Read-only viewers could not see an accepted AI summary on note detail because the page only rendered the editable AI summary panel, and version diffs ignored accepted-summary changes entirely. Note detail now renders a read-only summary display when needed, and diff output includes accepted-summary changes alongside body changes. Fixed in `b89a462`.
- The add-member form was visible to all org members even though only admins and owners can submit it, causing `member`-role users to see a form that would redirect them on use. The form is now hidden when `activeOrg.role === "member"` in the app shell sidebar.
